## 発端と動機（オーナーの言葉）

> 「Davinciに近い仕組みを利用した現像をスマホでもしたくて、最終アプリにできたらすごくいいなと思いつつ、まずは自分が使いたいだけって感じ。Lrだと色空間がDavinciみたいに選べないから、彩度と輝度を切り離して操作できないし、フィルムデンシティと呼ばれる色の密度の概念を取り入れた編集ツールが欲しいんだよね」

**オーナーが感じているLightroomの限界:**
- 色空間を選べない（Lrは独自空間固定）
- 彩度と輝度が切り離せない（HSLベースなので干渉する）
- 「フィルムデンシティ」の概念がない

**DaVinci Resolveのどこが好きか:**
- Log編集による輝度/色の分離
- 色空間を選んで作業できる感覚
- ノードベースの処理思想

## コア設計思想

### 「UIはLr、仕組みはDaVinci」
オーナーが明確に言語化した方針。
- 見た目・操作感: Lightroomのパネル型スライダーUI（直感的）
- 処理の内部構造: DaVinciのノードグラフ（見えないが確実に動いている）

### LABとHSVは「切り替え」ではなく「並列パネル」
最初「HSV空間の選択もあり得ると思うからその操作の切り替えができたらいいかも？いらないならいらないでOK」と悩んでいたが、
**ソラの提案として「切り替えでなく別パネルとして両方存在させる」方向に落ち着いた。**

| 空間 | 役割 | 強み |
|------|------|------|
| LAB | プライマリー補正 | 輝度↔彩度が完全独立。フィルムデンシティ再現に最適 |
| HSV | セカンダリー補正 | 色相回転・特定色の選択が直感的 |

## フィルムデンシティとは（設計者の定義）

オーナーの言葉:「フィルムデンシティと呼ばれる色の密度の概念」

数式的な意味：
- フィルムは露光量が上がるほど色が非線形に「積み上がる」
- 暗部は色が薄く、中間調で色が乗り、ハイライトで色が詰まる
- dye couplerの光学密度特性 = `density ∝ log(exposure)`

実装アプローチ:
- チャンネル別（R/G/B）カスタムトーンカーブで再現
- FilmConvert、FUJIFILMシミュレーターが近いことをやっている

## 処理パイプライン（フル設計）

```
写真入力（JPEG / RAW）
  ↓
色空間デコード（sRGB → Linear）
  ↓
[LABノード]         輝度(L) / 色(A/B) を完全独立操作        ← 無料
[HSVノード]         色相回転 / 彩度 / 明度（直感操作）       ← 無料
[カラーグレーディング] 3Wayカラーホイール（DaVinci式）        ← 課金
  ↓
[ゾーン分割]        輝度マスクでシャドウ/ハイライトを分離
  ├── [シャドウノード]    低輝度の彩度・粒子を独立操作        ← 課金
  └── [ハイライトノード]  高輝度の彩度・ハレーション           ← 課金
  ↓
[フィルムデンシティノード] チャンネル別トーンカーブ           ← 課金
[グレインノード]    全体 / ゾーン別粒子（シャドウに重く）     ← 課金
  ↓
出力エンコード（Linear → sRGB / Display P3）
  ↓
[テクスチャ合成]    紙質 / フィルム枠（最終合成パス）         ← 課金
  ↓
最終出力
```

### ハレーション（技術メモ）
- 高輝度部分を輝度マスクで抽出
- オレンジ〜赤寄りのガウシアンブラーをかけ「滲み」を作る
- 元画像に乗算合成
- フィルム乳剤層への光散乱を数式で再現

### テクスチャ合成（技術メモ）
- 最終パスでPNGをOverlay / Multiplyモードで重ねる
- 紙質: 紙のスキャンPNG
- フィルム枠: 35mm / 120mmのSVGまたはPNGオーバーレイ

## メディア管理機能（LrC的設計）

オーナーの言葉:
> 「PC版はLrCのようなプロジェクト単位での保管、データの選定に便利なように採用や除外フラグ、レート機能、カラーラベル、コレクション機能をつけたい」

### データ構造（IndexedDB）
```json
// photosテーブル（1行=1枚の写真）
{
  "path": "/Users/.../IMG_001.jpg",
  "flag": "pick" | "reject" | null,
  "rating": 0,
  "label": "red" | "yellow" | "green" | "blue" | "purple" | null,
  "collection_ids": ["vacation-2026", "favorites"],
  "recipe": {
    "lab": { "L": 12, "A": -5, "B": 3 },
    "hsv": { "H": 8, "S": 15, "V": -3 },
    "density_curve": [[0,0],[0.3,0.25],[1,1]],
    "film_grain": 0.4
  }
}
```

**LrCとの対応:**
| LrC機能 | 本アプリの実装 |
|---------|-------------|
| カタログ | IndexedDB database（プロジェクト単位） |
| フォルダ | File System Access APIでローカルフォルダを直接開く |
| 採用/除外フラグ | `flag: 'pick' | 'reject'` |
| レート | `rating: 0〜5` |
| カラーラベル | `label: 'red' | ...` |
| コレクション | collection テーブル（写真パスの仮想グループ） |

## プラットフォームとロードマップ

### なぜWebアプリ first か（重要な判断）
オーナーは最初「PC版デスクトップが欲しいと思ったけどwebアプリの方がいいのかな」と迷っていた。

**決定: Webアプリ first → Tauri desktop → mobile**

理由: Chromeの`File System Access API`（`window.showDirectoryPicker()`）でローカルフォルダを直接開けるため、ネイティブアプリに近い体験がブラウザで実現できる。処理（WebGL）はすべてクライアント側なのでサーバー負荷ゼロ。

| フェーズ | 形態 | 技術スタック |
|---------|------|------------|
| 1 | デスクトップWeb（Chrome） | Next.js + WebGL + File System Access API |
| 2 | デスクトップアプリ | Tauri（Next.jsをラップ・Electronより軽量） |
| 3 | モバイルWeb | レスポンシブ（カメラロール連携） |
| 4 | ネイティブアプリ | React Native（WebGLコア再利用） |

**サーバースペックの懸念について:**
オーナーが「サーバーのスペックが不安」と言っていたが、WebGLはユーザーの端末GPU処理なのでサーバー負荷ゼロ。現在のGrainDays環境でも問題ない。

## 長期ビジョン（プラットフォーム構想）

オーナーの言葉:
> 「将来的にはメディアというかユーザーが編集した写真、レシピをそのまま投稿するプラットフォーム兼エディターになるかなぁ。メインはエディターかも。」

**「レシピ」が鍵:**
現像設定（LAB/HSV/カーブなど）をJSONで持つ設計なら、そのまま「投稿コンテンツ」になる。他ユーザーがワンタップで同じ現像を再現できる。

VSCOのプリセット共有に近いが、パラメーターが開かれているため透明性が高く、学習コンテンツとしても機能する。
